# Architecture diagrams

[C4 model](https://c4model.com) by Simon Brown, written in [PlantUML](https://plantuml.com) using the [C4-PlantUML](https://github.com/plantuml-stdlib/C4-PlantUML) macros. Diagrams are committed as `.puml` source so reviewers can diff them; rendered SVG is regenerated by CI on push (TBD).

## Why C4 (not UML / arc42 / ad-hoc)

Same rationale as `golusoris/golusoris/docs/architecture/README.md` — sveltesentio mirrors golusoris's architecture-documentation standard:

- **Four levels** (Context → Container → Component → Code) matches how engineers actually think about systems — zoom in only where you need detail.
- **Plain text source** (PlantUML) means diff-friendly, code-review-friendly, no proprietary editor lock-in.
- **Adopted in [principles.md §2.7](../principles.md)** alongside ADRs as the architecture-documentation standard.

## Levels we maintain

| Level | File | Audience | When to update |
|---|---|---|---|
| **L1 Context** | [`context.puml`](context.puml) | Anyone (PMs, ops, security) | A new external system / actor enters the picture. |
| **L2 Container** | [`container.puml`](container.puml) | Engineers, ops | A new top-level `@sveltesentio/*` package lands. |
| **L3 Component** | `component-<package>.puml` | Engineers working on that package | Internal architecture changes worth recording. |
| **L4 Code** | (skipped) | — | Read the code; JSDoc / TSDoc + per-package `AGENTS.md` cover this. |

L4 is intentionally omitted. Code is the source of truth for code-level structure; diagrams duplicate and rot.
